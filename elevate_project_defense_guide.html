01. Tech Stack & System Architecture

Elevate Fitness is built using a modern Microservices Architecture. The project separates the application logic into three distinct subsystems to maximize efficiency: heavy database tracking is handled on the primary backend, fast UI interactions run in the browser, and compute-heavy Machine Learning regressions execute on a dedicated Python service.

💻

Frontend Subsystem

React.js SPA scaffolded with Vite. Renders real-time dashboards, interactive workout visualizers, and dietary planners. Integrates Google MediaPipe for local WebAssembly-based real-time video keypoint detection.

React.js Vite Vanilla CSS MediaPipe Tasks
⚡

Primary Backend Subsystem

Node.js Express server. Responsible for handling user sessions, processing secure Google OAuth logins, storing and fetching data, and acting as a secure rate-limited proxy to the Python ML server.

Node.js Express MongoDB JWT Auth
🐍

AI / ML Backend Subsystem

Python FastAPI microservice. Contains pre-trained multi-target machine learning models (XGBoost) and optimization algorithms. Handles biometric feature scaling, adaptive fatigue tuning, and shopping list generation.

FastAPI XGBoost Scikit-Learn Google Gemini
🛡️

Security & Resiliency Core

Defensive architecture features: Signed Double-Submit CSRF cookie validation, strict CORS lists, token-bucket endpoint rate limiters, downstream circuit breakers, and database self-healing routines.

CSRF Protection Circuit Breaker Rate Limiting Double-Signed Cookies

Examiner Checkpoint: Why a Microservices/Split Architecture?

"We decoupled the server into Express and FastAPI because Node.js is single-threaded and handles network I/O extremely fast, making it ideal for managing logins, OAuth, and web API routing. Python is slower for concurrency but is the undisputed industry standard for mathematical computation and ML. If we ran XGBoost and MediaPipe processing on a single Node thread, CPU-heavy tasks would block user requests, freezing the dashboard for all users."

02. User Profile & Request Lifecycle

Understanding the lifecycle of a user profile update is key to explaining how the frontend, backend, database, and AI engine interact. Here is exactly what happens when a user modifies their metrics:

Step 1
Form Input & Validation (React Frontend)
The user adjusts fields (e.g., age, weight, target goal, injuries, water intake) in ProfileSetup.jsx. The React form validates that inputs are within bounds (e.g. weight between 20-500kg, age between 1-120 years). If valid, it triggers an Axios call to PUT /api/python/profile/update.
Step 2
Authentication & Proxy Routing (Express Backend)
The Express server interceptor validates the user's session cookie. It strips any unsafe tags, attaches a tracing header x-request-id, and forwards the secure JSON payload to the Python FastAPI microservice (running on port 8000 or production Render container).
Step 3
Feature Engineering Pipeline (FastAPI Backend)
Python receives the payload in profile.py. The FeaturePipeline class normalizes the raw inputs, computes derived statistics like BMI, Recovery Score, and Progressive Overload delta, and hashes a 32-character Profile Fingerprint to detect if the core parameters changed.
Step 4
AI Invalidation & Model Execution
If the profile fingerprint differs from the database, the server invalidates old cached workout/meal structures. It executes the Multi-Output XGBoost Workout model and Multi-Target Nutrition model. The generated plan is passed to the Google Gemini service to formulate natural-sounding coaching texts.
Step 5
Database Persistence & UI Rerender
The finalized workout plan, meal configurations, and week metadata are saved in MongoDB using atomic write locks. FastAPI returns the generated plan to Express, which forwards it to React. The React Dashboard triggers a context state change, updating the UI charts and meal checklists instantly.

03. Core Features & Implementation Logic

Adaptive Workout Planner

Instead of hardcoding generic routines, the workout engine generates a tailored weekly split based on the user's specific constraints (equipment available, body injuries, target frequency, experience level).

  • Progression Engine: Evaluates past logged sessions. If a user logs successful workouts with consistent heart rates and completed sets, the system applies a progressive overload factor to automatically scale target reps or intensity.
  • Adaptive Modifier: Dynamically adjusts today's intensity based on the user's daily check-in (sleep hours, water intake, stress levels). If a user slept only 4 hours, their target sets/intensity are safely scaled down.
  • Injury Filtering: Hardcoded rules filter out exercises matching body issues. For example, if a user has a "Knee" injury, squat-based patterns are removed, replacing them with upper-body or leg-extension exercises.
Progression Engine Adaptive Modifiers Injury Shield

Deterministic Meal Optimizer

The meal planner predicts a user's daily TDEE (Total Daily Energy Expenditure) and splits it into precise breakfast, lunch, dinner, and snack targets.

  • Weekly Variety Tracker: Calculates penalty scores for meals. Points are deducted if a primary ingredient (e.g. paneer, chicken) or cuisine type is repeated too frequently within a 7-day window.
  • Portion Optimizer: Scales meal ingredients proportionally to hit exact caloric macros. Servings are snapped to standard increments (e.g. 10g, 50ml, whole numbers for chapatis/eggs) to ensure they are cookable in real life.
  • Dietary/Allergy Guardrails: Strictly blocklists foods containing user-defined allergens (e.g. Gluten, Nuts, Dairy) at the database query level, guaranteeing zero algorithmic contamination.
Variety Penalty Portion Snapping Allergy Guard

Real-time WebAssembly Pose Detector

Using the webcam feed, the system maps human joints in 3D to monitor exercise execution without sending video feeds to any server, preserving user privacy.

  • MediaPipe Integration: Loads lightweight WASM modules locally. Tracks 33 body coordinate points at 30+ FPS directly in the browser.
  • Hysteresis Rep Counting: Prevents false double-counts using threshold buffers. For example, during curls, the elbow angle must pass 150° (down state) and return past 60° (up state) to record one rep.
  • Form Feedback Engine: Compares joint alignments against safety ranges. Flags specific warnings like "Don't flare elbows too wide" during pushups to prevent rotator cuff strains.
Local WASM Execution 3D joint Mapping Form Safety Shield

NLP Generative Coaching Chatbot

To bridge the gap between raw numbers and a motivating human coach, the app integrates Google Gemini to translate outputs into interactive advice.

  • Coaching Context Injection: When chatting, the app injects user metrics (goals, workout plans, recent logs) into the LLM system prompt. The model knows you are weight training and recovering from an injury without you needing to type it.
  • Downstream Circuit Breaker: Keeps the app stable. If Gemini API reaches its quota limit, the python server trips the circuit open and serves local deterministic fallback guidelines, ensuring the chatbot never crashes.

04. Mathematical Models & Formulas

Our platform does not make random guesses. All caloric targets, recovery rates, progression schedules, and pose tracking angles are calculated using established clinical and geometric formulas.

1. Caloric Target & Energy Expenditure Models

First, the system calculates the user's Basal Metabolic Rate (BMR) using the clinically validated Mifflin-St Jeor equation:

$$BMR_{\text{Male}} = 10 \times W + 6.25 \times H - 5 \times A + 5$$ $$BMR_{\text{Female}} = 10 \times W + 6.25 \times H - 5 \times A - 161$$

The daily caloric requirement is determined by scaling BMR by an Activity Multiplier (resulting in TDEE), and then applying a Goal Multiplier:

$$TDEE = BMR \times \text{Multiplier}_{\text{Activity}}$$ $$\text{Calorie Target} = TDEE \times \text{Multiplier}_{\text{Goal}}$$
  • W: Weight in kilograms (kg)
  • H: Height in centimeters (cm)
  • A: Age in years
  • Activity Multipliers: Sedentary: 1.2, Light: 1.375, Moderate: 1.55, Active: 1.725, Extra Active: 1.9
  • Goal Multipliers: Weight Loss: 0.85, Fat Loss: 0.80, Gain Muscle: 1.10, Strength: 1.05, Maintenance: 1.0

2. Biometric Feature Engineering Models

The system derives a 4-factor Recovery Score from daily biometrics, which is then combined with consistency to yield a Readiness Score:

$$Recovery = \frac{(SleepScore/10) + (HydrationScore/10) + \frac{10 - Stress}{10} + \frac{10 - Fatigue}{10}}{4}$$ $$Readiness = Recovery \times 0.6 + Consistency \times 0.4$$

The rate of strength progression (overload delta) is adjusted dynamically using this formula:

$$\Delta_{\text{overload}} = \text{rate}_{\text{base}} \times \text{Adherence} \times Recovery \times \text{modifier}_{\text{exp}}$$ $$\text{Adherence} = 0.6 \times \text{Consistency} + 0.4 \times \min\left(1.0, \frac{\text{Streak}}{2 \times \text{DaysPerWeek}}\right)$$
  • Sleep/Hydration: Normalized values derived from hours slept and water consumed.
  • ratebase: Beginner: 0.03, Intermediate: 0.05, Advanced: 0.07
  • modifierexp: Beginner: 0.70, Intermediate: 1.00, Advanced: 1.15
  • Δoverload: Clamped strictly to the range \([0.0, 0.10]\).

3. Pose Tracking Trigonometric Model

To evaluate joint movements, we compute the angle \(\theta\) between three landmarks: joint vertex \(B(x_b, y_b, z_b)\), point \(A(x_a, y_a, z_a)\), and point \(C(x_c, y_c, z_c)\) using vector dot products in 3D coordinate space:

$$\vec{BA} = \langle x_a - x_b, y_a - y_b, z_a - z_b \rangle$$ $$\vec{BC} = \langle x_c - x_b, y_c - y_b, z_c - z_b \rangle$$ $$\theta = \arccos\left(\frac{\vec{BA} \cdot \vec{BC}}{\|\vec{BA}\| \times \|\vec{BC}\|}\right) \times \frac{180}{\pi}$$

4. Meal Optimization Scoring System

When generating daily meals, the engine tests candidates and scores them (out of 100 points) based on 7 dimensions. A higher score guarantees a better and safer meal plan:

Dimension Weight Mathematical Function / Rule
Macro Accuracy 30 Points Deducts based on percentage error from calorie target: \(Score = 30 - \min\left(30, \frac{|MealCal - TargetCal|}{TargetCal} \times 60\right)\)
Weekly Variety 20 Points Penalizes repetition of foods: Deducts \(min(20, \text{repeats} \times 0.3)\) for items flagged in the history sets.
Meal Completeness 15 Points Strict binary checklist: Deducts 15 points if Protein is missing; deducts 10 points if Carbs are missing in main meals.
Portion Realism 10 Points Deducts 3 points for each ingredient scaled to its absolute boundaries (clamped max/min portion limits).
Budget Preference 10 Points Calculates average food cost: Penalizes expensive items (High Budget = -8 pts) while keeping Low Budget at 0 penalty.
Availability Preference 10 Points Weighted sampling: Prefers common ingredients (Very Common = 4x weight) over rare ingredients (Rare = 0.25x weight).
Cuisine Rotation 5 Points Deducts 2 points for repeating the same regional cuisine in consecutive slots.

05. Machine Learning & Algorithm Workings

Elevate Fitness avoids hardcoded heuristics. The core fitness engine operates on custom-trained machine learning architectures. Let's look at how they work and how they process user data.

XGBoost MultiOutputRegressor

Primary Use: Predicting optimal sets, reps (low/high), rest times, and intensity based on biometrics.

XGBoost (Extreme Gradient Boosting) works by building sequential decision trees. Each new tree learns from the prediction errors (residuals) of the previous tree. We wrap the algorithm in a MultiOutputRegressor because target parameters are correlated (e.g., higher intensity implies longer rest times). Training them together captures these joint distributions.

XGBoost Nutrition Model

Primary Use: Predicting target calories per meal and total daily macros (proteins, carbs, fats).

Uses regression trees to output meal calorie distributions based on user activity levels and goals. A custom post-processing safety layer ensures that predicted meal calories strictly sum to the total daily calorie target derived from Mifflin-St Jeor. If the ML outputs drift, the layer scales them back proportionally.

Training, Hyperparameter Tuning & Evaluation

Here is the exact data preprocessing and training pipeline defined in multi_output_xgboost_model.py:

  • Feature Preprocessing: Encoders turn words into numbers: LabelEncoder is used for Experience (Beginner=0, Intermediate=1, Advanced=2) and Goal. StandardScaler normalizes values so numerical values are on a common scale.
  • Hyperparameter Tuning: We perform RandomizedSearchCV across a cross-validated grid testing parameters like maximum tree depth (max_depth between 3 and 10) and learning rate (learning_rate between 0.01 and 0.3).
  • Evaluation Metrics: The model quality is tracked using Mean Absolute Error (MAE) and Root Mean Squared Error (RMSE). We achieve an R² score of 94% - 96%, meaning 95% of workout parameter variances are correctly explained by our model.

ML Safety Guardrails & Clamps

AI regression models can sometimes hallucinate values outside normal limits (e.g. suggesting 12 sets or an intensity of 150%). To prevent this, the code implements Deterministic Overrides in the inference pipeline:

# Clamping predictions within absolute clinical boundaries
sets = np.clip(sets_pred, 1, 6)
reps_low = np.clip(reps_low_pred, 1, 20)

# Beginner safety override:
if experience == 'Beginner':
    intensity = np.minimum(intensity, 0.70) # Cap at 70% max intensity
    sets = np.minimum(sets, 4) # Cap at 4 sets max

During project evaluations, examiners will probe the technical decisions, codebase quality, and algorithmic implementations. Use this interactive Q&A grid to prepare your defense answers.

"Deep Learning models like Artificial Neural Networks are highly suited for unstructured data (images, video, text). However, for structured, tabular data (user profiles with age, height, weight, activity levels), decision-tree based models like XGBoost are mathematically superior."

"XGBoost builds sequential decision trees that optimize residuals. It requires far fewer parameters, trains in seconds rather than hours, uses a fraction of the RAM, does not suffer from vanishing gradients on small datasets, and achieves a test R² accuracy score of over 95% in predicting parameters."

Examiner Tip: Frame this as an engineering design choice. Focus on efficiency, cost, training time, and dataset suitability.

"No, we do not upload video frames to any server. Uploading high-definition video frames to a backend would require massive bandwidth, create latency bottlenecks, and violate user privacy."

"Instead, we load Google MediaPipe Tasks Vision WebAssembly (WASM) binaries locally in the user's browser. The client-side webcam feed is processed directly on the user's CPU/GPU. The system extracts 33 landmark coordinates as local JS objects, computes 3D trigonometry angles locally to track form, and only sends final rep-completion triggers to the backend."

Examiner Tip: Mention WebAssembly (WASM), User Privacy (GDPR/HIPAA compliance), and Zero-Latency edge computation.

"We implemented a strict, non-bypassable Deterministic Safety Layer (Guardrails) that wraps our ML output."

"Even if the regression model suggests extreme numbers due to outliers or data manipulation, the output is programmatically clamped to clinical bounds (e.g. sets are clamped to [1, 6] and daily protein is capped at 50% of total calories). Furthermore, for beginner users, the intensity is capped at 70% and sets are capped at 4, overriding any ML predictions."

Examiner Tip: Highlight that the system utilizes "AI for optimization" but "Hardcoded clinical constraints for safety."

"We utilize a two-level optimization check: Profile Fingerprinting and In-Memory Caching."

"When a user profile is fetched, the FastAPI server hashes the core biometric fields (age, weight, height, goal, injuries, experience) into a 32-character SHA-256 fingerprint. If this fingerprint matches the one stored in MongoDB, and the week has not changed, we skip the heavy ML execution entirely and load the plan from the cache. This reduces database writes and makes UI navigation instant."

Examiner Tip: Explain the SHA-256 hashing of profile metrics as a mechanism to minimize CPU load.

"In deterministic_meal_engine.py, the optimize_portions() function scales raw food portions to meet caloric macros. However, we snapping portions to specific intervals."

"For whole-unit food items (defined in metadata as 'piece' or 'unit' like eggs, bananas, chapatis), the quantity is rounded to the nearest integer: rounded_qty = max(1, round(qty)). For other foods (like chicken in grams, or oil in ml), we snap quantities to a food-specific portion_step parameter (e.g. snapping oats to the nearest 10g step), preventing uncookable fractional weights."

Examiner Tip: Show that the database contains metadata columns specifying unit types and step size boundaries for each ingredient.

"We implement defense-in-depth across the tech stack:"

  • CSRF Shield: We enforce double-submit signed cookies via csrf-csrf middleware. State-mutating routes (POST, PUT, DELETE) must match the signed token header, preventing unauthorized requests from external sites.
  • NoSQL Injection Guard: User IDs are validated against ObjectId.is_valid() before executing Mongoose queries. Unsafe parameter types are rejected at the router level.
  • Secure Auth: User sessions are managed using HttpOnly cookies, preventing client-side XSS attacks from reading user JWT tokens.
