An end-to-end audit, architectural analysis, and defense handbook documenting the implementation, mathematical algorithms, and machine learning components of the Elevate ecosystem.

1. System Architecture & Tech Stack

Technical Stack Defense

Elevate is built upon a resilient Microservice-inspired Multi-Tier Architecture. The system splits concerns cleanly between user interaction, database persistence, and computational data science.

Component Technology Used Role in System Engineering Rationale & Why We Selected It
Frontend React.js (Vite) Dynamic UI rendering, Client-side routing, Real-time camera tracking. Vite provides near-instant HMR (Hot Module Replacement) and compilation, bypassing Webpack overhead. React handles stateful components like webcam feeds and graphs efficiently.
Gateway & Auth Node.js (Express) Google OAuth, Session token verification, Security layers (CSRF/Signed Cookies), MongoDB Gateway. Asynchronous event-loop architecture makes Node exceptionally fast for I/O operations, DB queries, and proxying. Single-threaded non-blocking I/O is ideal for gateway traffic.
AI & Analytics Python (FastAPI) XGBoost models prediction, portion optimization algorithms, MediaPipe tracking calculations. FastAPI utilizes Python's type hints to auto-generate OpenAPI documentation and runs asynchronous requests using Uvicorn. Python is critical due to its scikit-learn and XGBoost ecosystem.
Database MongoDB Document storage of user profiles, daily metrics logs, and historical logs. NoSQL allows a flexible schema. As a user completes more check-ins or logs custom exercises, MongoDB stores the dynamic attributes without the schema migration downtime of SQL.

2. Core Features Catalog

1. AI-Driven Workout Prescription

Automatically calculates a user's target parameters: Sets, Rep Ranges (Low/High), Rest Interval (seconds), and Intensity Percentage. Rather than using generic templates, it feeds biometric parameters (Age, Weight, Experience, Goal) into an XGBoost Regressor model trained on expert fitness layouts.

XGBoost Regression Custom Feature Pipeline

2. Real-Time Pose Tracking & Form Verification

Using the device camera, the frontend streams joint landmarks in real-time. The system tracks 33 critical coordinates via Google MediaPipe. In-app algorithms calculate joint angles (elbow flexion for pushups, hip/knee flexion for squats) to track repetitions, identify breakdown in form, and suggest corrective adjustments.

Google MediaPipe Joint Trigonometry

3. Portions Optimizer & Meal Planner

Generates a complete 7-day meal plan categorized by 4 meal slots (Breakfast, Lunch, Dinner, Snack). Features a Portion Scaling Algorithm that scales ingredients to snap to logical boundaries (e.g. integer eggs, whole chapatis, 50g steps for rice) and dynamically recommends food swaps that maintain caloric limits.

Deterministic Optimization Indian Food Database

4. Weekly Adaptive Biometric Modifiers

The system tracks daily check-ins (water consumed, sleep hours, workout completion). Before generating a new plan, the adaptive_modifier.py script evaluates biometric averages to detect dehydration or sleep deficits. It adjusts intensity down by up to 25% or triggers a deload week to prevent injury.

Biometric Normalizer CNS Fatigue Clamps

5. Deterministic Safety Guardrails

Machine Learning models can output wild recommendations (hallucinations or mathematical drifts). To prevent injuries, we built a strict safety layer on top of the model outputs.

Beginner Caps

Any user whose profile states "Beginner" is immediately capped at an intensity of 0.70 (70%) and a maximum of 4 sets, overriding the ML model output.

Age-Based Clamps

Seniors (65+) are clamped to a max intensity of 0.70, sets capped at 3, rest forced to a minimum of 90 seconds, and high-impact exercises are filtered out.

Allergen Filtering

The meal engine filters out foods that match user allergies (egg, milk, soy, nuts, gluten) using regex boundary matches on ingredients before candidates are scored.

4. Mathematical Models & Equations

Biometrics & Energy Calculations

Elevate implements standard clinical physiology equations to normalize user baseline attributes.

1. Basal Metabolic Rate (BMR) - Mifflin-St Jeor Equation

BMR = 10 * Weight(kg) + 6.25 * Height(cm) - 5 * Age(y) + S
Where the constant S acts as a gender offset:
  • Male: S = +5
  • Female: S = -161
  • Other: S = -78

2. Total Daily Energy Expenditure (TDEE) & Calorie Target

TDEE = BMR * Activity_Multiplier
Calorie_Target = TDEE * Goal_Multiplier
  • Activity Multipliers: Sedentary (1.2), Lightly Active (1.375), Moderately Active (1.55), Active (1.725), Very Active (1.9).
  • Goal Multipliers: Weight Loss (0.85), Fat Loss (0.80), Muscle Gain (1.10), Strength (1.05), Maintenance (1.00).

Adaptive Progress Overload Math

These mathematical formulas are implemented deterministically in progression_engine.py to manage overload transitions.

3. Adherence Score

Adherence = 0.60 * Completion_Pct + 0.40 * Streak_Factor
Streak_Factor = min(Streak_Days / (2 * Days_Per_Week), 1.0)
Determines the consistency coefficient. A user who performs all scheduled workouts achieves an Adherence score of 1.0.

4. Sleep & Hydration Normalization (Biometric Normalizer)

Sleep_Score = 8.0 + ((Sleep_Hours - 7.0) / 2.0) * 2.0 [for 7 to 9 hours]
Hydration_Score = 7.0 + ((Water_Liters - 2.0) / 1.5) * 3.0 [for 2.0L to 3.5L]
Translates raw physical checks into a unified 1-10 scale. Non-linear mapping is applied for excess values (e.g. sleep > 12 hours lowers the recovery score due to grogginess).

5. Central Nervous System (CNS) Readiness Score

Readiness = 0.40 * Sleep_Norm + 0.35 * Fatigue_Norm + 0.25 * Stress_Norm
Recovery_Factor = Readiness / 100.0
Calculates systemic recovery. Where:
  • Fatigue Norm: (10.0 - Fatigue_Level)
  • Stress Norm: (10.0 - Stress_Level)

6. Progression Delta Equation

Delta = Base_Rate * Adherence * Recovery_Factor * Experience_Mod
Calculates the rate of overload adjustments.
  • Base Rates: Beginner (0.03), Intermediate (0.05), Advanced (0.07).
  • Experience Modifiers: Beginner (0.70), Intermediate (1.00), Advanced (1.15).
  • Clamped to range: [0.0, 0.10]. Applied directly to intensity or sets.

Computer Vision Angle Math

Calculates joint flexion vectors in real-time camera frames inside pose_tracker.py.

7. Landmark 3-Point Trigonometric Angle

v1 = [a_x - b_x, a_y - b_y], v2 = [c_x - b_x, c_y - b_y]
Radians = atan2(v2_y, v2_x) - atan2(v1_y, v1_x)
Angle = |Radians * 180 / PI|; if Angle > 180 then 360 - Angle
Where b represents the joint center (e.g. elbow) and a, c are the surrounding skeletal connections (shoulder and wrist).

5. XGBoost & Machine Learning Engine

Why XGBoost Over Neural Networks?

If the examiner asks why a Neural Network wasn't used, here is the answer:

"Neural Networks are powerful for unstructured data like images and voice files. However, for structured, tabular user data (age, weight, height, goal indices), decision-tree based models perform significantly better. XGBoost builds sequential decision trees where each tree corrects the errors of its predecessor. It is computationally efficient, runs comfortably on a low-cost CPU backend, does not overfit on small/medium tabular datasets, and allows us to extract clear feature importances to explain recommendations."

Multi-Output & Multi-Target Regressors

Standard regressions predict a single variable (e.g., just calories). Elevate implements Multi-Output Regression pipelines to preserve correlations between predicted variables:

Workout Parameter Model

Predicts 5 output variables simultaneously:

  • sets (volume indicator)
  • reps_low (minimum rep target)
  • reps_high (maximum rep target)
  • rest_seconds (CNS recovery window)
  • intensity (percentage of 1RM)
Nutrition Target Model

Predicts 7 outputs simultaneously:

  • breakfast_calories
  • lunch_calories
  • dinner_calories
  • snack_calories
  • protein_total (g)
  • carbs_total (g)
  • fats_total (g)

Hyperparameter Tuning & Training Pipeline

The model is trained via train_model.py and train_meal_model.py. The training steps are detailed below:

1
Preprocessing: Categorical features (Experience, Goal, Activity Level) are normalized using LabelEncoder. Values are scaled using StandardScaler to ensure variables with larger absolute values (e.g. Weight = 110) do not overpower variables with smaller ranges (e.g. Age = 20).
2
Tuning: Utilizes RandomizedSearchCV. Instead of checking every combination, it randomly samples from a defined distribution grid of hyperparameters to find optimal settings within a limited time budget.
3
Tuned Parameters: Max Depth of trees (3 to 10), Learning Rate/shrinkage (0.01 to 0.3), Subsample ratio (0.6 to 1.0), and L1/L2 regularization (alpha and lambda) to prevent tree complexity from causing overfitting.

6. Beginner to Intermediate Progression

The Progression Pathway

How does a user progress from a beginner to an intermediate/advanced athlete? In Elevate, this is handled through both profile updates and adherence-based adjustments.

1. Initial Onboarding

User joins as "Beginner". Base progression rate is set to 3%. Capped to 4 sets max, and intensity capped at 70%. Rest periods are longer to build recovery baseline.

2. Accumulating Consistency

As user logs workouts, the Adherence Score builds up. If streak hits 10 days, they receive a volume upgrade (+1 set). If streak hits 30 days, they unlock exercise progressions.

3. Upgrading Experience

Once their logged history count increases, the system suggests a profile transition to "Intermediate". This increases the base rate to 5%, sets cap to 5, and lowers rest periods.

🔄
Plateau Breaking: If the detect_plateaus() function identifies 3 consecutive sessions without weight or repetition increases on a specific movement pattern (e.g. Squat), the system flags a plateau. The engine then sets the progression method to plateau_breaker, suggesting a variation swap (e.g. swap Goblet Squats for Jump Squats) to stimulate muscle adaptation.

7. Examiner Q&A (Viva Handbook)

Q1: Why did you split the system into Node.js and Python? Why not use Python for everything? +
This is an example of the Separation of Concerns design pattern. Node.js is built on an event-driven, non-blocking I/O model, making it exceptionally fast for handling user authentication (Google OAuth), managing API routing, and serving data from MongoDB. However, Python is historically slow for high-concurrency web requests, but is the industry standard for mathematical computation and machine learning (XGBoost, MediaPipe). By splitting the architecture, each server is highly optimized for its specific task. If the AI model consumes 100% CPU during training, user login and profile endpoints on Node.js remain unaffected.
Q2: Is your AI recommendation engine just a wrapper around the Google Gemini API? +
No. We built a hybrid intelligence system. The mathematical predictions (sets, reps, rest, intensity, calories, macros) are computed entirely locally on our backend using our trained XGBoost Regressor model and deterministic physiological algorithms. Google Gemini is used solely as a Natural Language Generation (NLG) layer. It takes our raw mathematical parameters and translates them into motivating, readable coaching summaries so that the user receives human-like guidance.
Q3: What security measures did you implement to protect the user database and backend? +
We implemented a comprehensive security suite on the Node.js API Gateway:
  • CSRF Protection: Enforced via the csrf-csrf package using the Signed Double-Submit Cookie pattern. Mutating endpoints (POST/PUT) are rejected if the header token doesn't match the encrypted cookie.
  • Secure HttpOnly Cookies: JWT session tokens are stored in HttpOnly cookies with sameSite: 'lax' (or 'none' in production) and secure: true flags. This makes them inaccessible to client-side scripts, completely preventing Cross-Site Scripting (XSS) token theft.
  • Secure Headers: We manually inject standard Helmet headers like X-Content-Type-Options: nosniff, X-Frame-Options: DENY, and strict HSTS in production.
Q4: What happens if the Google Gemini API key is missing or the external API service goes down? +
We implemented a Circuit Breaker pattern in backend-python/app/circuit_breaker.py using the tenacity package. If the Gemini API fails 5 consecutive times, the circuit breaker trips OPEN. Any subsequent chatbot requests are short-circuited and bypassed immediately without waiting for network timeouts. The system falls back gracefully to offline heuristics (hardcoded motivational coaching tips) until the recovery timeout of 60 seconds passes, after which it enters the HALF-OPEN state to test if the API is healthy again.
Q5: How does the camera-based pose tracker work mathematically? +
The frontend accesses the device camera via WebRTC and feeds frames to Google MediaPipe's skeletal landmark pipeline. This identifies 33 coordinates on the body in a 3D coordinate system (x, y, z normalized between 0 and 1). We extract the 3 coordinates for the active joint (e.g. Hip, Knee, and Ankle for squats). We treat these coordinates as vectors and calculate the angle using atan2:
Radians = atan2(Ankle_y - Knee_y, Ankle_x - Knee_x) - atan2(Hip_y - Knee_y, Hip_x - Knee_x)
This is converted to degrees. If the angle goes below 90 degrees (deep squat), the state updates to "down". Once the angle extends back past 170 degrees, the state updates to "up" and the rep count increments.
Q6: How does the Portion Optimizer work? Why not just divide calories by average food calories? +
Simply dividing calories would recommend unrealistic portions like "1.42 eggs" or "0.87 chapatis". The Portion Optimizer scales the baseline recipe items proportionally first, then rounds them to the nearest portion step (e.g., eggs round to integers, rice rounds to the nearest 50g, oil to 5ml). It then clamps each rounded value between the metadata's portion_min and portion_max limits to prevent ridiculous quantities. Finally, it runs a scoring algorithm to verify the resulting macro combination is as close to the target profile as possible.
